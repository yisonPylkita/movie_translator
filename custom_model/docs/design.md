# Custom EN→PL Subtitle Translation Model — Design Document

**Date:** 2026-04-04
**Status:** Design phase

## Overview

Build a purpose-built English-to-Polish translation model for anime and movie subtitles, trained on a high-quality parallel corpus generated by LLM translation and refined by human review. The model uses a context-aware architecture with three input channels: show metadata, episode metadata, and a sliding dialogue window.

This document covers two separable workstreams:
1. **Dataset** — collecting, generating, and curating high-quality parallel translation data with rich metadata
2. **Model** — a context-aware translation architecture trained on this data

These are intentionally decoupled. The dataset is a long-lived asset that can be reused as model architectures improve. The model can be retrained at any time on improved data.

## Part 1: The Dataset

### Design principles

- The dataset stores **everything that might be useful for future models**, not just what the current architecture needs
- Data provenance is tracked (who translated it, was it verified)
- The schema supports iterative refinement (LLM first pass → human review → corrections)
- Show and episode metadata is stored alongside translation pairs

### Storage format: JSON files as source of truth, JSONL as training export

The dataset has two layers:

1. **Source of truth** — human-readable JSON files, one per episode/movie, organized in directories. This is what we maintain, review, and version-control. It preserves hierarchical structure (show metadata, episode context, ordered pairs).

2. **Training export** — machine-readable JSONL (one record per translation pair) generated from the source files by a build script. This is what the training script consumes. It flattens the hierarchy, attaches sliding window context, and shuffles for training.

Why JSON files and not a database: transparency. You can browse, diff, and review individual episode files. A database is opaque — you can't git diff it, and the queries you'd run ("export all pairs for training") are trivially solved by reading files from a directory. At our scale (1-2M lines across thousands of files), a database solves problems we don't have.

Why per-episode files and not one giant JSON: at 1M+ pairs a single file becomes unwieldy — slow to parse, impossible to diff, one syntax error corrupts everything. Per-episode files are 10-50KB each, independently valid, independently reviewable.

### Directory structure

```
dataset/
  dataset.json              # version, creation date, global statistics
  shows/
    berserk-1997.json       # show-level metadata
    frieren-2023.json
    ...
  episodes/
    berserk-1997/
      s01e01.json           # episode metadata + translation pairs
      s01e02.json
      ...
    frieren-2023/
      s01e01.json
      ...
  movies/
    spirited-away-2001.json # movie-level metadata + translation pairs

training/                   # generated by build script, not hand-edited
  train.jsonl               # flat file, one translation pair per line
  val.jsonl                 # validation split (held-out episodes)
  stats.json                # pair counts, source distribution, etc.
```

The `training/` directory is a build artifact — regenerated from `dataset/` whenever the source data changes. The build script reads all episode JSONs, looks up show metadata, assembles the sliding dialogue window for each pair, and writes flat JSONL lines.

### Training export format (JSONL)

Each line in `train.jsonl` is a self-contained training example:

```jsonl
{"source": "March!", "target": "Naprzód!", "show_context": "Dark fantasy anime. Medieval European setting. Major battle scene.", "dialogue_prev": ["Why is Griffith so late anyway..."], "dialogue_next": ["For me, a true friend..."], "characters": ["Griffith", "Guts", "Casca"]}
{"source": "What was that?", "target": "Co powiedziałeś?", "show_context": "Dark fantasy anime. Medieval battle. Casca fights Lord Adon.", "dialogue_prev": ["So useless... under that woman's thumb."], "dialogue_next": ["What will you do now?"], "characters": ["Casca", "Adon", "Guts"]}
```

The training script doesn't need to know about episodes or shows — it streams self-contained records. This is the abstraction boundary between data curation and model training.

A formal JSON Schema will validate the source files, ensuring every contributor (human or LLM) produces data in the correct format.

### Data sources for English subtitles

- **Personal video library** — already have the subtitle tracks
- **OpenSubtitles corpus** — millions of EN subtitle files, freely available for research
- **Anime-specific subtitle databases** — AnimeSub, Kitsunekko
- **Fetched by movie-translator itself** — the existing fetch pipeline already downloads EN subtitles

### Polish translation sources (in quality order)

1. **Human professional translation** — highest quality, most expensive, use for validation set
2. **LLM translation + human review** — Claude translates, human verifies and corrects. Best cost/quality ratio for bulk data.
3. **LLM translation only** — Claude Sonnet for high-quality lines, Claude Haiku for bulk. Cheaper but unverified.
4. **Existing Polish subtitles** — from subtitle databases. Variable quality, but free and abundant. Good for mixing with synthetic data to prevent model collapse.

### Data generation pipeline

```
Phase 1: Collect English subtitles
  ├── Extract from personal video library
  ├── Download from OpenSubtitles for target shows
  └── Parse into structured format (strip ASS/SRT formatting)

Phase 2: Enrich with metadata
  ├── Show metadata: fetch from TMDB/MAL APIs or write manually
  ├── Episode summaries: fetch from TMDB/MAL or generate with LLM
  ├── Character lists: extract from subtitles (proper noun detection) + manual
  └── Genre/tone classification: from TMDB tags

Phase 3: Generate Polish translations
  ├── Use Claude API (Batch API for 50% cost reduction)
  ├── Prompt includes: show description, episode summary, character list,
  │   surrounding dialogue lines for context
  ├── Generate in batches of 10-20 lines per API call
  └── Store with provenance (source, date, model version)

Phase 4: Quality assurance
  ├── Automated checks (length ratio, untranslated words, character names)
  ├── Human review of sampled lines (flag and correct errors)
  ├── Mark verified pairs
  └── Mix with human-translated data from Polish subtitle databases
```

## Part 2: The Model Architecture

### Overview

An encoder-decoder translation model with three input channels:

1. **Context encoder** — processes show description + episode summary once, produces a fixed representation that conditions all translations for that episode
2. **Dialogue encoder** — processes a sliding window of surrounding dialogue lines plus the current source line
3. **Translation decoder** — generates the Polish translation, cross-attending to both the context representation and the dialogue encoding

```
                    ┌─────────────────────┐
                    │   Context Encoder    │
                    │                      │
                    │  Show: dark_fantasy, │
                    │  medieval, Berserk   │
                    │  Episode: battle,    │
                    │  Casca vs Adon       │
                    └──────────┬───────────┘
                               │ context representation
                               │ (computed once per episode)
                               ▼
┌──────────────────────────────────────────────────┐
│                Dialogue Encoder                   │
│                                                   │
│  [prev-2] I'll find a way to kill him!            │
│  [prev-1] So useless... under that woman's thumb. │
│  [current] What was that?  ◄── line to translate  │
│  [next-1] What will you do now?                   │
│  [next-2] There is no ground behind you.          │
└──────────────────────┬───────────────────────────┘
                       │ dialogue encoding
                       ▼
              ┌─────────────────┐
              │    Decoder      │
              │  (cross-attends │
              │   to both)      │
              │                 │
              │  → Co powiedz-  │
              │    iałeś?       │
              └─────────────────┘
```

### Context encoder details

- Processes show metadata + episode summary as a single text block
- For anime episodes: show description + episode summary
- For movies: show/movie description only (no episode context)
- The output is a fixed-length representation (e.g., pooled encoder states)
- **Computed once per episode/movie** and cached — no per-line cost
- Can be a lightweight transformer (2-4 layers) or a shared encoder with the dialogue encoder

### Dialogue encoder details

- Sliding window of N previous + M future lines around the current line
- The current line is marked with a special token so the decoder knows which line to translate
- Previous and future lines provide disambiguation context

### Open question: dialogue window content

**Option A: English-only window**

All lines in the window (past, current, future) are in English. The model learns to use English context to produce better Polish.

```
[prev-1] EN: So useless... under that woman's thumb.
[current] EN: What was that?
[next-1] EN: What will you do now?
→ PL: Co powiedziałeś?
```

Advantages:
- **Full batch parallelism** — all lines can be translated independently since no line depends on another line's translation output
- Simpler training — all context is in one language
- At inference time, all context is immediately available (we have the full subtitle file)

Disadvantages:
- The model can't maintain Polish-language coherence across lines (e.g., consistent use of formal/informal register)
- Previous Polish output could help resolve gender/number agreement

**Option B: Mixed-language window (autoregressive)**

Previous lines include their Polish translations (model's own output). Future lines are English-only.

```
[prev-1] EN: So useless... | PL: Tak bezużyteczni...
[current] EN: What was that?
[next-1] EN: What will you do now?
→ PL: Co powiedziałeś?
```

Advantages:
- The model can maintain Polish coherence (register, pronouns, style)
- Previous translations provide strong signals for continuation

Disadvantages:
- **Sequential processing** — must translate line N before line N+1, no parallelism
- Error propagation — a bad translation feeds into context for subsequent lines
- 2-5x slower at inference time
- More complex training (need to simulate autoregressive context during training)

**Recommendation to evaluate:** Start with Option A for the initial model. If coherence across lines is visibly worse than expected, try Option B as a follow-up experiment. Option A is simpler to implement, train, and deploy, and the context encoder (show/episode metadata) may provide enough coherence signal to make Option B unnecessary.

### Why single-pair focus matters

This model is exclusively EN→PL. No multilingual models — every parameter must be spent on this single language pair.

A model has a fixed number of parameters. Each parameter stores learned patterns. In a multilingual model like NLLB-200 (200 languages, 600M params), those parameters are shared across all language pairs — the same attention heads handle Japanese→French and English→Polish. Each pair gets a fraction of the model's capacity.

A single-pair model dedicates 100% of capacity to EN→PL:
- **Larger effective vocabulary** — all tokens cover English input and Polish output, no budget spent on Japanese/Korean/Arabic scripts
- **Better Polish morphology** — more capacity for diacritics (ą, ć, ę, ł, ń, ó, ś, ź, ż), case endings, and gender agreement
- **Deeper idiom learning** — can memorize more EN→PL specific patterns instead of cross-language generalizations

Evidence: OpusMT en-pl at 211M parameters scores comparably to NLLB-200 at 600M (3x larger) on en-pl benchmarks — because most of NLLB's extra parameters serve other pairs.

### Parameter budget and hardware targeting

Parameters are declared up front via architecture choices (layers, hidden size, vocab size). The same 211M-parameter model can be trained on 1 language pair or 200 — the parameter count doesn't change. But the **depth of knowledge per pair** is inversely proportional to the number of pairs trained.

Target hardware: Apple Silicon Macs (M-series, 8-64GB unified memory). The model runs alongside video processing (FFmpeg, OCR), so it shouldn't dominate memory.

| Model size | FP16 memory | INT8 (CTranslate2) | Fits target? |
|---|---|---|---|
| 100M params | ~200MB | ~100MB | Easily, even on 8GB Macs |
| **200-250M params** | **~400-500MB** | **~200-250MB** | **Yes — matches current Allegro footprint** |
| 500M params | ~1GB | ~500MB | Yes but heavier |
| 1B params | ~2GB | ~1GB | Tight on 8GB Macs |

**Target: 200-250M parameters.** This matches the current Allegro model (211M, 800MB on disk in FP32), so the infrastructure doesn't change. With INT8 quantization via CTranslate2, the model fits in ~200MB and runs fast on any Apple Silicon Mac.

### Base model selection

| Option | Params | License | Rationale |
|---|---|---|---|
| **OpusMT en-pl (fine-tune)** | 211M | Apache-2.0 | Dedicated EN→PL specialist. Every parameter is already trained for this pair. Our codebase supports it. Requires architecture modification for dual encoder. |
| **T5-base (fine-tune for EN→PL)** | 220M | Apache-2.0 | English-only foundation. More flexible text-to-text architecture — easier to add context via prefix or cross-attention. Larger community and tooling for modifications. |
| **MarianMT (train from scratch)** | ~200M | Apache-2.0 | Clean slate, fully custom. Requires more training data but gives complete control over architecture and vocabulary. |

**Recommendation:** Start with **OpusMT en-pl** (fine-tune) for Phase 1-2 since it already has strong EN→PL knowledge and we can measure gains from our data immediately. For Phase 3 (full dual-encoder architecture), consider **T5-base** as it's more amenable to architectural modifications like adding a separate context encoder with cross-attention.

For the context encoder, start with a **shared encoder** (same weights as the dialogue encoder, different input) to keep parameter count down. Split into separate encoders only if experiments show benefit.

### Training plan

**Phase 1: Baseline (no context)**
- Fine-tune T5-base on the parallel corpus with no metadata
- Establish quality baseline
- Compare against Allegro and Apple

**Phase 2: Add context prefix (approach 1 from earlier research)**
- Prepend genre/show tags to source text
- Retrain and measure improvement
- Cheapest way to test whether context helps

**Phase 3: Full architecture (dual encoder)**
- Add dedicated context encoder
- Train with show + episode metadata
- Add sliding dialogue window
- Compare Option A vs Option B for window content

**Phase 4: Optimization**
- Quantize with CTranslate2 (INT8)
- Benchmark latency on Apple Silicon
- Target: faster than current Allegro (13s/episode), ideally matching Apple (~4s)

## Open Questions

1. **Option A vs Option B** for dialogue window content (see above)
2. **How much human review** is needed for the training data? Pilot experiment: generate 1000 lines, have a human review, measure error rate, decide if bulk human review is worth the cost.
3. **Character name handling** — should the model learn to preserve character names from the character list in show metadata, or should we keep the placeholder protection layer on top?
4. **Evaluation metric** — BLEU is standard but correlates poorly with subtitle quality. Consider human evaluation on a held-out test set of ~500 lines from diverse shows.
5. **Training data diversity** — how many different shows/genres are needed to prevent overfitting? Aim for at least 50 shows across 5+ genres.
